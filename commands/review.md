# 브랜치 변경사항 코드 리뷰

## 작업 순서

1. **현재 브랜치 정보 확인**

   - `git branch --show-current` 명령으로 현재 브랜치 이름 확인
   - `git log origin/main..HEAD --oneline` 명령으로 main 브랜치 대비 커밋 목록 확인

2. **변경된 파일 목록 확인**

   - `git diff origin/main...HEAD --name-status` 명령으로 변경된 파일 목록 확인
   - 추가(A), 수정(M), 삭제(D) 파일 분류

3. **변경 내용 상세 분석**

   - `git diff origin/main...HEAD` 명령으로 전체 diff 확인
   - 각 파일별 변경사항 분석

4. **전체 코드베이스 대상 확인**

   - 변경사항에 반영되어야 하지만 변경되지 않은 부분이 코드베이스에 남아있는지 확인
   - 전체 코드베이스의 코드 형태와 변경사항이 잘 어우러지는지 확인
   - 프로젝트에 제공된 코드 분석 도구(formatter, linter 등)를 사용해서 코드베이스에 문제가 없는지 확인

5. **코드 리뷰 수행**
   아래 관점에서 변경사항을 리뷰해주세요:

```md
### 🔍 코드 품질

- 코드 가독성 및 명명 규칙
- 중복 코드 여부
- 함수/메서드 크기 및 책임
- 논리적 무결성(Soundness/Correctness) 및 최소한의 코드베이스 유지 확인

### 🔄 불필요한 직접 구현 탐지

- 프로젝트에 이미 설치된 라이브러리로 대체 가능한 직접 구현이 있는지 확인
- 프로젝트 내 기존 유틸리티/헬퍼를 활용하지 않고 중복 구현한 부분이 있는지 확인
- 의존성 파일(package.json, Cargo.toml, requirements.txt 등) 확인하여 이미 설치된 관련 패키지가 있는지 확인

### 🐛 잠재적 버그

- 엣지 케이스 처리
- 널/언디파인드 처리
- 에러 핸들링

### 🔒 보안

- 민감한 정보 노출 여부
- 입력값 검증
- 인증/인가 관련 이슈

### ⚡ 성능

- 불필요한 연산
- 메모리 누수 가능성
- N+1 쿼리 등 DB 관련 이슈

### 📝 문서화

- 주석의 적절성
- API 문서화 여부
- README 업데이트 필요성
```

6. **리뷰 결과 출력**
   다음 형식으로 리뷰 결과를 정리해주세요:

```md
## 📋 브랜치 리뷰 요약

**브랜치명:** [브랜치 이름]
**커밋 수:** [N개]
**변경 파일 수:** [N개]

## ✅ 잘된 점

- [긍정적인 부분들]

## ⚠️ 개선 제안

- [파일명:라인] 설명

## 🚨 반드시 수정 필요

- [심각한 이슈가 있다면]

## 💡 추가 제안

- [선택적 개선사항]
```

## 참고사항

- 비교 대상 브랜치가 `main`이 아닌 경우, 사용자에게 확인 후 적절한 브랜치로 변경
- diff가 너무 큰 경우 파일별로 나눠서 분석
- 변경사항이 없는 경우 사용자에게 알림
