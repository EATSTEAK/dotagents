# 선택형 코드 리뷰

## 리뷰 대상 설정

- 리뷰 대상: `$ARGUMENTS`
- 인자가 없으면 `branch`를 사용
- 지원 형식:
  - `/review` 또는 `/review branch` → 현재 브랜치 diff 리뷰
  - `/review staged` → staged diff 리뷰
  - `/review file path/to/file` → 특정 파일 전체 리뷰
- `branch`는 현재 구현 기준 리뷰입니다. 기본 비교 대상은 `origin/main...HEAD`입니다.
- `file` 선택 시 반드시 파일 경로를 함께 입력하세요.

## 작업 순서

1. **리뷰 대상 해석**

   - `$ARGUMENTS`가 비어 있으면 `branch`로 간주
   - 첫 번째 인자가 `branch`, `staged`, `file` 중 무엇인지 확인
   - `file`인데 파일 경로가 없으면 사용자에게 경로를 다시 요청
   - 지정되지 않은 값이면 지원 형식을 안내하고 다시 확인

2. **대상별 변경 범위 확인**

   - `branch`
     - `git branch --show-current` 명령으로 현재 브랜치 이름 확인
     - `git log origin/main..HEAD --oneline` 명령으로 main 브랜치 대비 커밋 목록 확인
     - `git diff origin/main...HEAD --name-status` 명령으로 변경된 파일 목록 확인
     - `git diff origin/main...HEAD` 명령으로 전체 diff 확인
   - `staged`
     - `git diff --cached --name-status` 명령으로 staged 파일 목록 확인
     - `git diff --cached` 명령으로 staged diff 전체 확인
   - `file`
     - 대상 파일이 존재하는지 확인
     - 파일 전체 내용을 읽고 구조와 구현을 파악
     - 필요하면 해당 파일의 관련 참조 지점도 함께 확인

3. **전체 코드베이스 대상 확인**

   - 변경사항 또는 대상 파일에 반영되어야 하지만 주변 코드베이스에 남아있는 문제점이 있는지 확인
   - 전체 코드베이스의 코드 형태와 리뷰 대상이 잘 어우러지는지 확인
   - 프로젝트에 제공된 코드 분석 도구(formatter, linter 등)를 사용할 수 있으면 함께 확인

4. **코드 리뷰 수행**
   아래 관점에서 변경사항 또는 파일 전체를 리뷰해주세요:

```md
### 🔍 코드 품질

- 코드 가독성 및 명명 규칙
- 중복 코드 여부
- 함수/메서드 크기 및 책임
- 논리적 무결성(Soundness/Correctness) 및 최소한의 코드베이스 유지 확인

### 🔄 불필요한 직접 구현 탐지

- 프로젝트에 이미 설치된 라이브러리로 대체 가능한 직접 구현이 있는지 확인
- 프로젝트 내 기존 유틸리티/헬퍼를 활용하지 않고 중복 구현한 부분이 있는지 확인
- 의존성 파일(package.json, Cargo.toml, requirements.txt 등) 확인하여 이미 설치된 관련 패키지가 있는지 확인

### 🐛 잠재적 버그

- 엣지 케이스 처리
- 널/언디파인드 처리
- 에러 핸들링

### 🔒 보안

- 민감한 정보 노출 여부
- 입력값 검증
- 인증/인가 관련 이슈

### ⚡ 성능

- 불필요한 연산
- 메모리 누수 가능성
- N+1 쿼리 등 DB 관련 이슈

### 📝 문서화

- 주석의 적절성
- API 문서화 여부
- README 업데이트 필요성
```

5. **리뷰 결과 출력**
   다음 형식으로 리뷰 결과를 정리해주세요:

```md
## 📋 리뷰 요약

**리뷰 대상:** [branch | staged | file]
**대상 설명:** [브랜치명 / staged diff / 파일 경로]
**커밋 수:** [N개, 해당 시]
**변경 파일 수:** [N개, 해당 시]

## ✅ 잘된 점

- [긍정적인 부분들]

## ⚠️ 개선 제안

- [파일명:라인] 설명

## 🚨 반드시 수정 필요

- [심각한 이슈가 있다면]

## 💡 추가 제안

- [선택적 개선사항]
```

## 참고사항

- `branch`의 비교 대상 브랜치를 바꾸려면 사용자에게 먼저 확인
- diff가 너무 큰 경우 파일별로 나눠서 분석
- `staged`에 변경사항이 없거나 `file` 경로가 잘못된 경우 바로 사용자에게 알림
- 변경사항이 없는 경우 사용자에게 알림
